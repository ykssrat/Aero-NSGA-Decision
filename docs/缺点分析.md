# OptiSelect — 工程缺点与局限性分析

---

## 1. 数据硬编码问题严重

目前所有组件的参数库（例如 `PC_Problem.Components_WithCase.Param1` 这类包含 `[成本, 性能, 属性3, 属性4, 属性5]` 的矩阵）被**硬性编码**写死在了主函数 `computer.m` 内。
- **缺点**：这种做法对于原型验证是足够快捷的。但在实际投入业务使用时，真实组件的数据（包括各维度的成本评估表、不同供应商性能参数等）通常需动态更新。工程师甚至可能需要导入千级别的元件库数据，硬编码方式会让代码快速膨胀且无法维护。
- **改进建议**：应重构数据读取模块，支持从外部 Excel 表格、CSV 报表或直接连接 MySQL/SQLite 数据库等方式动态加载 `Components_*` 矩阵数据。

## 2. 交互方式受限于控制台

当前所有筛选条件和配置需求都是通过 MATLAB `input()` 指令在命令行界面同步、阻断式采集的。
- **缺点**：缺少一个能够持久化保存用户配置的方案。如果用户想要“基于上一次输入的筛选条件仅修改其中某一项参数”，他必须重新跑一遍程序并且将几十个交互问题重新回答一遍，用户体验差。
- **改进建议**：引入 JSON 或 XML 等配置文件（如 `config.json`），允许程序自动读取默认配置；或者采用 MATLAB App Designer 构建轻量级的参数配置 UI 界面。

## 3. 多目标决策权重的固化

在 `TOPSIS_Selection.m` 调用处（主文件 `computer.m` 中），分配给 TOPSIS 的权重 `weights = [0.3, 0.7]`（对应 30%成本，70%性能）被直接写死。
- **缺点**：多目标优化的核心价值在于能够让决策者在不同场景下应用不同的业务偏好。这（比如在经费紧张的项目里，成本可能需要调整为 80% 的权重）如果每次都要改代码，就不符合“通用选型系统”的设计初衷。
- **改进建议**：应将这个权重的赋予提升为用户交互输入项之一，或是作为配置文件中的全局变量开放给业务终端。

## 4. 特殊兼容性业务逻辑的强耦合

脚本 `CalculateCompatibility_WithCase.m` 中内置了十分具象的领域逻辑（如第55行的“电源功率估算” `cpu_power = [65, 95, 125, 150, 200]`，以及大量硬编码的搭配条件：高端 CPU 配基础散热器降分等）。
- **缺点**：这使得原本“通用化”的 NSGA-II + TOPSIS 框架与“台式电脑主机装机”这一特定业务产生了极强的强耦合。如果要把这套工程平移去解决航天器的模块选型，那么这个兼容性校验脚本就必须完全推翻重写。
- **改进建议**：应当将这些约束提取为“惩罚函数库”或“规则引擎配置文件”，让算法本身保持纯净，仅保留对约束判定接口的调用。

## 5. 项目模块引用的独立性问题

当前的目录结构虽然已重组，将所有的算法脚本移至 `/scripts` 子目录中以提升根目录清爽度。
- **缺点**：但在 MATLAB 中，如果没有在工作空间全局添加 `scripts` 路径，那么当用户试图写个新的上层调用脚本时，就会报“未定义函数”的错误（因为默认不会跨文件夹检索）。
- **改进建议**：主入口程序应当第一步就显式包含如下代码以保证环境健壮性：
  ```matlab
  % 获取当前执行文件所在目录并自动将 scripts 加入 MATLAB 检索路径
  scriptPath = fullfile(fileparts(mfilename('fullpath')), 'scripts');
  addpath(scriptPath);
  ```
